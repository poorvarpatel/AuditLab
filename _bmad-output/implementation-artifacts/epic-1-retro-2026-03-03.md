## What Went Well

### Technical Wins

- **Background context pattern (Story 1-2):** Establishing `viewContext` as READONLY and all writes on `newBackgroundContext()` was a breakthrough. This pattern carried cleanly through Stories 1-3 and 1-4. No main-thread blocking from Core Data saves.
- **Layered architecture respected:** Persistence layer stayed clean; Stores as single callers; no View → Persistence leakage. Architecture compliance was strong across all stories.
- **Repository protocol:** DocumentRepositoryProtocol provided a clear contract. Protocol extensions (queue, settings, history) were additive and consistent.
- **Test coverage:** In-memory PersistenceController for tests; DocumentRepositoryTests grew from initial coverage to 22 tests including edge cases (optional Booleans, negative duration, restart persistence, folder filter).

### Process Wins

- **Code review effectiveness:** Every story had code review fixes. Recurring catches: force unwraps → guard/cast, error handling, File List completion, edge-case tests. The review process prevented production bugs.
- **Story sequencing:** 1-1 → 1-2 → 1-3 → 1-4 order worked well. Each story built on the previous; no rework from ordering.
- **Previous Story Intelligence:** Story files carried forward context (background context pattern, repository init) so each story started with correct assumptions.

### Team Collaboration

- **Consistent patterns:** Once Story 1-2 established background context + PersistenceController reference, Stories 1-3 and 1-4 followed the same pattern without drift.
- **Documentation in stories:** Dev Notes, Completion Notes, and File List provided traceability. Code review fixes were documented in Change Log.

## Challenges and Growth Areas

### Technical Challenges

1. **Optional Core Data Booleans (Story 1-4):** QueueEntry `incApp` and `incSum` are optional in Core Data. Code initially used force unwraps; review caught and fixed with `entry.incApp ?? true`. **Lesson:** Core Data optional attributes need explicit default handling at mapping boundaries.
2. **Repository API extension (Story 1-4):** `updateFolderName` was added to DocumentRepository without being in the original story scope. Documented as deviation in Completion Notes. **Lesson:** When extending repository API mid-epic, document the deviation and ensure it's justified.
3. **AppSet / AppSettings split:** Voice, rate, appearance in Core Data (AppSettings); skipAsk, figBg, wps in UserDefaults (AppSet). Left as-is per story scope. **Lesson:** Consolidation could be a future Epic 7 cleanup if it causes friction.

### Process Challenges

1. **File List completeness:** Code reviews repeatedly flagged incomplete File List. **Lesson:** Add "File List complete" to story completion checklist before marking review-ready.
2. **Scope clarification (Story 1-4):** Epic/PRD described "migration from current state" but story assumed no prior users—no migration runner. Scope was simplified and worked for brownfield. **Lesson:** When epic wording differs from implementation assumption, call it out in story so it's intentional.

### Recurring Review Feedback

- Force unwraps → use guard/cast and custom error types (DocumentRepositoryError)
- Persistence failures → log in DEBUG, don't swallow
- Partial failure recovery → e.g. QueueStore.remove(atOffsets:) resyncs via loadEntries() on failure
- Test edge cases → nil voiceIdentifier, empty updateQueueOrder, negative duration

---

## Key Insights

1. **Background context from day one:** Establishing the readonly viewContext + background writes in Story 1-2 prevented main-thread blocking and set the pattern for the rest of the epic. Worth the refactor cost.
2. **Code review catches what unit tests miss:** Edge cases (optional Booleans, nil handling) surfaced in review, not in initial tests. Review is a quality gate.
3. **Brownfield + no users = simpler migration:** Story 1-4 assumed no prior app usage. No migration runner, no guard keys. This simplified scope and reduced risk. If real users exist later, a separate migration story would be needed.
4. **Store mapping preserves view types:** LibStore, FoldStore, QueueStore map Document/Folder/QueueEntry → PaperRec/FoldRec/QItem. Views needed minimal change. Mapping at Store boundary worked well.
