## What Went Well

### Technical Wins

- **LibStore as single owner of library state:** Add, delete, loading (isAddingDocument, loadingPackId), and recs all lived in LibStore. Views bound only to LibStore; no View → Persistence. Architecture boundaries held across all six stories.
- **Loading and empty states (NFR-U1, NFR-U2):** Story 2-1 established isAddingDocument and overlay; 2-3 added libraryEmptyState; 2-4 used loadingPackId for detail; 2-6 verified and standardized. One loading indicator per operation, within 500 ms via synchronous state updates.
- **Cascade delete and referential integrity:** Story 2-5 leveraged Core Data delete rules (Document → DocumentFolder cascade; QueueEntry/HistoryItem nullify). Pack cache cleanup moved to reloadFromContext after save, avoiding race conditions. DocumentDeleteCascadeTests and code review caught nil doc.identity handling.
- **Accessibility consistency:** Empty state, Add PDF button, delete action, document detail, and loading overlays got identifiers and labels (library-empty-state, library-add-pdf-loading, document-detail-loading, etc.). UI tests use A11y constants.

### Process Wins

- **Previous Story Intelligence in story files:** Dev Notes and Project Structure Notes carried context (LibStore patterns, Core Data rules, file touchpoints). Each story built on the last without re-discovering.
- **Code review as quality gate:** Reviews repeatedly caught: File List completeness (2-5), missing user-facing error handling (2-5 deleteError), pack cache timing (2-5), optional/unwrap safety (2-6 DocumentDeleteCascadeTests). Action items (2-6) created for test strength and documentation.
- **Story sequencing:** 2-1 → 2-2 → 2-3 → 2-4 → 2-5 → 2-6 order worked. Add → view list → view detail → delete → polish empty/loading. No rework from ordering.

## Challenges and Growth Areas

### Technical Challenges

1. **File List vs git scope (2-5, 2-6):** Multiple stories modified the same app files (LibraryView, LibStore, PaperDetailView, etc.). Code review had to distinguish “this story’s changes” from “other stories’ changes.” Story 2-6 File List included DocumentDeleteCascadeTests (a 2-5 test) after fixing a regression—blurring story boundaries. **Lesson:** Keep File List strictly to files changed for that story; document cross-story fixes in Completion Notes only.
2. **Weak loading-state assertion (2-6):** testDocumentDetailShowsLoadingState passes if loading OR content appears. When pack is cached or loads fast, the test doesn’t assert that loading UI was shown. **Lesson:** Either force uncached load in test and assert loading view, or document that the test is lenient and manual verification covers loading.
3. **Optional/unwrap in tests (2-6):** DocumentDeleteCascadeTests had doc.identity! after refresh; failed when identity was nil. Fixed by capturing paperId before delete. **Lesson:** In tests that delete or refresh, capture IDs and key state before operations that can invalidate managed objects.

### Process Challenges

1. **Code review findings as action items:** Story 2-6 used “Review Follow-ups (AI)” with checkboxes. Marking the story “done” without resolving those items can leave technical debt untracked. **Lesson:** Either fix MEDIUM/HIGH in review or keep story in-progress until follow-ups are done; treat action items as real commitments.
2. **Manual verification record:** Stories mark manual verification [x] but rarely record date or evidence. **Lesson:** One-line note in Completion Notes (e.g. “Manual verification: empty state, add-PDF overlay, detail loading — 2026-03-05”) improves traceability.

### Recurring Review Feedback

- File List: include every modified/created file; separate “this story” from “other stories.”
- User-facing errors: every failure path (add error, delete error) must set Store state and show Alert/banner.
- Test robustness: avoid force unwraps on Core Data objects after refresh; prefer capturing IDs before destructive operations.
- Accessibility: identifiers and labels on empty state, loading overlays, and actions; use constants in UI tests.

---

## Key Insights

1. **LibStore as single source of truth for library UX:** isAddingDocument, loadingPackId, recs, addError, deleteError in one place kept views simple and prevented duplicate loading/error UI. Same pattern should apply to Queue and History when they get similar flows.
2. **Empty and loading states pay off early:** Establishing them in 2-1 and 2-3 made 2-6 mostly verification and small enhancements. Investing in NFR-U1/NFR-U2 from the first story reduced rework.
3. **Code review catches integration and docs:** File List accuracy, error handling, and test safety were caught in review, not in initial implementation. Keeping review before “done” is a quality gate.
4. **Story boundaries matter:** Mixing 2-5 test fixes into 2-6 File List confused ownership. Clear “this story only” File List and explicit notes for cross-story fixes keep retrospectives and audits clear.
